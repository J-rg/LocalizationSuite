BLFileInterpreter

The primitive abstract superclass for all file interpretations.

Superclass: NSObject
Declared In: BLFileInterpreter.h

Discussion

Basically, a file interpreter is a one-shot object. You have it created, set it up and use it - afterwards it is gone. Subclasses should override the _interpreteFile: method to actually do the heavy lifting.

A technique sometimes needed is interpreter forwarding: If an interpreter needs to use a different one, it can just instantiate it and have it's contents forwarded. A reason for this might be that the data has only been converted to a different format, which is then to be interpreted. The procedure is simple as follows: In the overridden _interpreteFile: you should instantiate the different interpreter of the needed type and then send it a _setForwardsToInterpreter: message with self as argument. After that the others _interpreteFile: should be called directly.

Discussion

actualPathForHashValueGeneration:

Return the subpath of a path to be used for hash generation. 

- (NSString *)actualPathForHashValueGeneration:(NSString *)path;
Discussion

This allows subclasses to specify a different path for hash generation. This is especially needed if the file path is a bundle, because hash values can only be calsulated for files. The default implementation returns the given path, which should be fine for most implementations.

autotranslateKeyObject:usingObjects:forLanguage:

Autotranslate the given key object. 

- (void)autotranslateKeyObject:(BLKeyObject *)keyObject 
        usingObjects:(NSArray *)objects forLanguage:(NSString *)language;
Discussion

By running through the given objects, which must be an array of BLKeyObjects, the key's value for language is compared. If a exact match is found, all localized values that keyObject has not value for will be copied. The result might be a key object merged from multiple key objects. This method is mainly for internal use.

hashValueForFile:

Calculates the hash for a given file path. 

- (NSString *)hashValueForFile:(NSString *)path;
Discussion

The result should be a string that always changes if the contents of a file change. The change date should be ignored, as only content changes should be relevant. Default uses the MD5 hash algorithm to generate the hash.

init

Designated initializer for subclasses. 

- (id)init;
Discussion

Subclasses should override this if they need to make changes other than to the default set of options, e.g. if you need some internal structure.

interpreteFile:

Interpretes a file into a newly created file object. 

- (BLFileObject *)interpreteFile:(NSString *)path;
Return Value

A file object of the right type with the scanned contents or nil, if an error occurred.

Discussion

As most simple use, this method scans the file's path for a language folder, creates an according file object and scans the contents of the file to that file object, which is then returned.

interpreteFile:intoObject:

Interpretes a file at the given path into the given file object. 

- (BOOL)interpreteFile:(NSString *)path intoObject:(BLFileObject *)object;
Return Value

YES if the file was imported, NO if no import is required or an error happened.

Discussion

The path is scanned for an .lproj folder and it's language is used for interpretation. However, if no lanuage is found nothing is imported.

interpreteFile:intoObject:withLanguage:

Interpretes a file for a given language into a given file object. 

- (BOOL)interpreteFile:(NSString *)path intoObject:(BLFileObject *)object 
        withLanguage:(NSString *)language;
Discussion

This method first checks whether the file will be imported and then sets up the import environment, calling _interpreteFile: in the subclass'es implementation. This is the central method that is called by the methods above and might be an major entry point for external callers. However, it should not be overridden by subclassers as a lot of setup and teardown happens here. _interpreteFile: is the way to do this. If you need to internally use a different interpreter, see the class description on how to do interpreter forwarding.

interpreteFileObject:inDocument:withLanguage:

In an document context, this importes a localized version of a file object. 

- (BOOL)interpreteFileObject:(BLFileObject *)object 
        inDocument:(NSDocument <BLDocumentProtocol> *)document 
        withLanguage:(NSString *)language;
Return Value

YES if the file was imported, NO if no import is required or an error happened.

Discussion

The documents path creator is consulted for the full path of the given file object, then this method forwards to interpreteFile:intoObject:withLanguage:.

willInterpreteFile:intoObject:

Determines whether an import is needed or not. 

- (BOOL)willInterpreteFile:(NSString *)path intoObject:(BLFileObject *)object;
Return Value

YES if the file will be imported, NO if an import call will have no change.

Discussion

The default implementation compares the change date with the last import and also checks whether the file hash, which is computed using hashValueForFile:, has changed. If a date and a hash change is detected, YES is returned, NO otherwise.

Determines whether an import is needed or not. 

- (BOOL)willInterpreteFileObject:(BLFileObject *)object 
        inDocument:(NSDocument <BLDocumentProtocol> *)document 
        withLanguage:(NSString *)language;
Discussion

As a convenience method, this consults the document's path creator and then returns the result of willInterpreteFile:intoObject:.
